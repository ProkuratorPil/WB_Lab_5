### 1.2. Модуль кеширования

**Файл:** `app/core/cache.py`

Реализован сервис `CacheService` — единая точка работы с Redis.

**Алгоритм подключения:**
1. При импорте модуля создаётся экземпляр `CacheService`.
2. В `__init__` вызывается `_connect()`, который пытается установить соединение с Redis.
3. Если соединение не удалось — `_connected = False`, приложение продолжает работать без кеша (graceful degradation).
4. Перед каждой операцией вызывается `_ensure_connection()` — попытка восстановить соединение.

**Методы сервиса:**

| Метод | Описание | Безопасность |
|-------|----------|-------------|
| `get(key)` | Получает значение, десериализует JSON | При ошибке возвращает `None` |
| `set(key, value, ttl)` | Сериализует в JSON, сохраняет с TTL | При ошибке возвращает `False` |
| `delete(key)` | Удаляет ключ | При ошибке возвращает `False` |
| `delete_by_pattern(pattern)` | Использует `SCAN` + `UNLINK` для безопасного удаления по паттерну | Не блокирует Redis |

**Ключевые особенности:**
- Используется `decode_responses=True` — данные возвращаются как строки.
- `socket_connect_timeout=3` и `socket_timeout=3` — защита от зависания.
- `json.dumps(value, default=str)` — сериализация объектов (например, UUID → строка).

### 1.3. Кеширование списков пользователей

**Файл:** `app/services/user_service.py`, метод `get_all_active()`

**Алгоритм:**
```
1. Формируем ключ: f"wp:users:list:page:{pagination.page}:limit:{pagination.limit}"
2. Проверяем кеш: cache_service.get(cache_key)
3. Если Cache Hit:
   - Возвращаем cached["users"], cached["total"]
   - Запрос к БД НЕ выполняется
4. Если Cache Miss:
   - Выполняем запрос к БД (COUNT + SELECT с OFFSET/LIMIT)
   - Сериализуем результаты в JSON
   - Сохраняем в кеш: cache_service.set(cache_key, {"users": users_data, "total": total}, ttl=300)
   - Возвращаем данные
```

**Инвалидация:**
- `POST /users/` (create) — вызывает `cache_service.delete_by_pattern("wp:users:list:*")`
- `PUT/PATCH /users/{id}` (update) — вызывает удаление списка, детали и профиля
- `DELETE /users/{id}` (delete) — вызывает удаление списка, детали и профиля

### 1.5. Управление Access токенами через JTI в Redis

**Файл:** `app/core/jwt.py`

Access токен теперь содержит поле `jti` (JWT ID):
```python
payload = {
    "sub": str(user_id),
    "type": "access",
    "jti": jti,  # <-- добавлено
    "exp": expires_at,
    "iat": datetime.now(timezone.utc)
}
```

**Файл:** `app/routers/auth_router.py`, функция `_save_tokens()`

**Алгоритм сохранения JTI:**
```
1. Извлекаем access_jti из tokens
2. Вычисляем TTL: int(jwt_manager.access_expires_delta.total_seconds()) = 900 сек (15 мин)
3. Формируем ключ: f"wp:auth:user:{user_id}:access:{access_jti}"
4. Сохраняем в Redis: cache_service.set(redis_key, "valid", ttl=900)
```

**Файл:** `app/core/dependencies.py`, функция `_check_access_jti_in_redis()`

**Алгоритм проверки:**
```
1. После валидации подписи JWT извлекаем payload
2. Получаем user_id = UUID(payload["sub"]) и jti = payload["jti"]
3. Формируем ключ: f"wp:auth:user:{user_id}:access:{jti}"
4. Проверяем наличие в Redis: cache_service.get(key)
5. Если значение None (ключ удалён или истёк) → 401 Unauthorized
6. Если значение есть → продолжаем обработку
```

**Файл:** `app/routers/auth_router.py`, endpoint `/auth/logout`

**Алгоритм отзыва сессии:**
```
1. Получаем access_token из cookies
2. Декодируем токен через jwt_manager.decode_token()
3. Извлекаем jti из payload
4. Удаляем ключ из Redis: cache_service.delete(f"wp:auth:user:{user_id}:access:{jti}")
5. Отзываем refresh токен в БД
6. Удаляем профиль из кеша
7. Очищаем cookies
```

**Файл:** `app/routers/auth_router.py`, endpoint `/auth/logout-all`

**Алгоритм отзыва всех сессий:**
```
1. Отзываем все токены в БД через revoke_all_user_tokens()
2. Удаляем ВСЕ JTI пользователя: cache_service.delete_by_pattern(f"wp:auth:user:{user_id}:access:*")
3. Удаляем профиль из кеша
4. Очищаем cookies
```

## 2. Ответы на контрольные вопросы

### 2.1. В чем разница между стратегиями Cache-Aside и Write-Through?

**Cache-Aside (Lazy Loading):**
- Приложение само управляет кешем.
- При чтении: проверяет кеш → при Miss загружает из БД → сохраняет в кеш.
- При записи: обновляет БД → инвалидирует (удаляет) кеш.
- **Плюсы:** Простота, гибкость, кеш обновляется только при необходимости.
- **Минусы:** Первый запрос после инвалидации медленный (Cache Miss).
- **Где используется:** В данной работе — `UserService.get_all_active()`, `auth_router.whoami()`.

**Write-Through:**
- Данные одновременно пишутся в БД и в кеш.
- При чтении: всегда берём из кеша.
- **Плюсы:** Кеш всегда актуален, нет Cache Miss.
- **Минусы:** Запись медленнее (два вызова), избыточное кеширование неиспользуемых данных.
- **Где НЕ используется:** В данной работе выбран Cache-Aside, так как он проще и более гибок для REST API.

### 2.2. Что такое "Thundering Herd problem" и как кеширование может помочь или усугубить её?

**Определение:**
Thundering Herd ("топот стада") — ситуация, когда кеш сбрасывается, и множество параллельных запросов одновременно обращаются к БД для перестроения кеша, создавая пиковую нагрузку.

**Как кеширование помогает:**
- При Cache Hit запросы обслуживаются из Redis (быстро, без нагрузки на БД).

**Как кеширование усугубляет:**
- Если кеш сбрасывается (инвалидация), и сразу приходит 1000 запросов — все они пойдут в БД.

**Решения:**
1. **Cache Warming** — предварительное заполнение кеша после инвалидации.
2. **Mutex/Lock** — блокировка: только один запрос строит кеш, остальные ждут.
3. **Staggered TTL** — случайный разброс TTL, чтобы ключи не истекали одновременно.

**В данной работе:** Graceful degradation помогает — если Redis недоступен, запросы идут напрямую в БД без падения приложения.

### 2.3. Почему не рекомендуется использовать команду `KEYS *` в продакшене на больших объёмах данных?

**Проблема:**
- `KEYS *` сканирует ВСЕ ключи в БД за один раз.
- Время выполнения — O(N), где N — количество всех ключей.
- На миллионах ключей команда может выполняться секунды или минуты.
- Redis — однопоточный: во время выполнения `KEYS` все остальные запросы блокируются.

**Решение:**
- Использовать `SCAN` — итеративный перебор с небольшими порциями.
- В данной работе: `cache_service.delete_by_pattern()` использует `SCAN` + `UNLINK`.

**Код:**
```python
# app/core/cache.py
cursor = 0
while True:
    cursor, keys = self._client.scan(cursor=cursor, match=pattern, count=100)
    if keys:
        self._client.unlink(*keys)
    if cursor == 0:
        break
```

### 2.4. Как обеспечить согласованность данных между БД и кешем при одновременной записи?

**Проблема:**
- Поток A обновляет БД, но ещё не инвалидировал кеш.
- Поток B читает устаревшие данные из кеша.

**Стратегии:**

1. **Cache-Aside с инвалидацией (используется в работе):**
   - Сначала обновляем БД (COMMIT).
   - Затем удаляем ключи из кеша.
   - Следующий запрос построит кеш заново.
   - **Риск:** между COMMIT и delete может быть чтение устаревших данных.

2. **Update-in-Place:**
   - Обновляем БД и сразу обновляем кеш (не удаляем).
   - Более сложно, но кеш всегда актуален.

3. **Transactional Outbox:**
   - Запись в БД + событие в outbox таблице.
   - Отдельный процесс читает outbox и обновляет кеш.

**В данной работе:** используется подход 1 (Cache-Aside с инвалидацией). Для учебного проекта это допустимо. В продакшене можно добавить distributed lock.

### 2.5. Зачем нужен TTL у ключей кеша, если мы реализуем инвалидацию при записи?

**Причины:**

1. **Защита от утечек памяти:** Если инвалидация по какой-то причине не сработала (баг, сбой, race condition), TTL гарантирует автоматическое удаление.

2. **Предотвращение устаревания:** Если данные изменились в обход API (напрямую в БД), кеш сам очистится через TTL.

3. **Управление памятью:** Redis работает в RAM. TTL позволяет автоматически освобождать память.

4. **Graceful degradation:** Если Redis перезапускается без персистентности, TTL предотвращает хранение бесконечно устаревших данных.

**В данной работе:**
```python
# app/core/config.py
CACHE_TTL_DEFAULT: int = 300  # 5 минут

# app/core/cache.py
expire = ttl if ttl is not None else settings.CACHE_TTL_DEFAULT
self._client.setex(key, expire, serialized)
```

### 2.6. Почему хранение JTI Access токена в Redis позволяет реализовать мгновенный Logout, несмотря на Stateless природу JWT?

**Проблема Stateless JWT:**
- JWT содержит всю информацию внутри себя (sub, exp, и т.д.).
- Сервер может проверить подпись и срок действия без обращения к БД.
- Но если токен украден — его нельзя отозвать до истечения срока (например, 15 минут).

**Решение через JTI + Redis:**
- В JWT добавляется уникальный идентификатор `jti`.
- При генерации токена `jti` сохраняется в Redis с TTL = сроку жизни токена.
- При каждом запросе сервер проверяет: есть ли `jti` в Redis?
- При logout ключ `jti` удаляется из Redis.
- Следующий запрос с этим токеном: подпись валидна, срок не истёк, но `jti` в Redis отсутствует → `401 Unauthorized`.

**Где реализовано:**
- Генерация JTI: `app/core/jwt.py`, `create_access_token()`
- Сохранение JTI: `app/routers/auth_router.py`, `_save_tokens()`
- Проверка JTI: `app/core/dependencies.py`, `_check_access_jti_in_redis()`
- Удаление JTI: `app/routers/auth_router.py`, `logout()` и `logout_all()`

### 2.7. Какие данные из лабораторной работы №3 (токены, сессии) безопасно кешировать, а какие нет?

**Безопасно кешировать:**

| Данные | Где | Почему безопасно |
|--------|-----|-----------------|
| JTI (JWT ID) | `wp:auth:user:{id}:access:{jti}` | Это просто UUID, без полезной нагрузки. Без самого токена бесполезен. |
| Профиль пользователя | `wp:users:profile:{id}` | Публичные данные (username, email). Без паролей. |
| Список пользователей | `wp:users:list:*` | Публичные данные в рамках авторизованного доступа. |

**НЕ безопасно кешировать:**

| Данные | Почему опасно |
|--------|--------------|
| Пароли (даже хеши) | Утечка хеша = компрометация аккаунта. Требование лабораторной: "Запрещено хранение паролей в кеше". |
| Полные JWT токены | Если Redis скомпрометирован — злоумышленник получает живой токен. Требование: "Запрещено хранение полных токенов доступа в открытом виде". |
| Refresh токены | Аналогично, дают право на выпуск новых Access токенов. |
| Персональные данные (ПДн) | email, телефон — в кеше допустимо только в рамках авторизованного доступа и с TTL. |

**В данной работе:**
- ✅ В кеше хранится только JTI, не полный токен.
- ✅ Пароли никогда не попадают в кеш.
- ✅ Профили кешируются без чувствительных полей.

### 2.8. Как повлияет перезапуск контейнера Redis на работу приложения?

**Сценарий 1: С персистентностью (AOF/RDB) — используется в работе**

В `docker-compose.yml`:
```yaml
command: redis-server --requirepass ${REDIS_PASSWORD} --appendonly yes
volumes:
  - wp_labs_redis:/data
```

- `--appendonly yes` включает AOF: каждая операция записи добавляется в файл.
- При перезапуске Redis восстанавливает данные из AOF.
- Влияние на приложение:
  - JTI токенов сохранятся → сессии останутся активными.
  - Кеши пользователей сохранятся → производительность не упадёт.
  - Но если данные в БД изменились во время отключения Redis — кеш может быть устаревшим до истечения TTL.

**Сценарий 2: Без персистентности**

- Все данные в Redis хранятся только в RAM.
- При перезапуске все ключи теряются.
- Влияние на приложение:
  - Все сессии станут недействительными (JTI удалены) → все пользователи получат `401`.
  - Кеши пропадут → первые запросы будут медленными (Cache Miss), но приложение не упадёт.
  - Graceful degradation сработает: при отсутствии Redis приложение продолжает работать напрямую с БД.

**В данной работе:** AOF включён, volume подключён — данные сохраняются между перезапусками.

### 2.9. Что такое сериализация и зачем она нужна при сохранении объектов в Redis?

**Определение:**
Сериализация — процесс преобразования объекта (словарь, список, модель Pydantic) в формат, который можно сохранить или передать (строку, байты).

**Почему нужна в Redis:**
- Redis — хранилище ключ-значение, где значение — строка или байты.
- Нельзя напрямую сохранить Python-объект (`dict`, `list`, `UUID`, `datetime`).
- Нужно преобразовать в строку → сохранить → при чтении преобразовать обратно.

**В данной работе:**
```python
# app/core/cache.py
# Сериализация (запись):
serialized = json.dumps(value, default=str)
self._client.setex(key, expire, serialized)

# Десериализация (чтение):
raw = self._client.get(key)
return json.loads(raw)
```

- `json.dumps(value, default=str)` — преобразует объект в JSON-строку.
- `default=str` — обработчик для нестандартных типов (UUID, datetime → строка).
- `json.loads(raw)` — преобразует JSON-строку обратно в Python-объект.

**Альтернативы:**
- `pickle` — бинарная сериализация Python-объектов (быстрее, но небезопасна — может выполнить произвольный код).
- `msgpack` — бинарный формат (компактнее JSON).
- В данной работе выбран `json` — универсальный, безопасный, человекочитаемый формат.

### 2.10. Как префиксы ключей помогают в управлении кешем?

**Формат префиксов в работе:**
```
wp:module:entity:identifier
```

**Примеры:**
- `wp:users:list:page:1:limit:10` — список пользователей
- `wp:users:detail:123e4567-...` — конкретный пользователь
- `wp:users:profile:123e4567-...` — профиль
- `wp:auth:user:123e4567-...:access:abc-def` — JTI токена

**Преимущества:**

1. **Избежание конфликтов:** Если в Redis используются другие сервисы, префикс `wp:` гарантирует уникальность.

2. **Массовая инвалидация:** Можно удалить все ключи модуля:
   ```bash
   UNLINK wp:users:*      # удалить всё, связанное с пользователями
   UNLINK wp:auth:user:*  # удалить все JTI
   ```

3. **Мониторинг и отладка:** По префиксу сразу понятно, что за данные:
   ```bash
   KEYS wp:users:*   # посмотреть все кеши пользователей
   KEYS wp:auth:*    # посмотреть все сессии
   ```

4. **Организация пространства имён:** Логическая структура ключей повторяет архитектуру приложения.

**В данной работе:**
```python
# Кеш списка
cache_key = f"wp:users:list:page:{pagination.page}:limit:{pagination.limit}"

# Кеш профиля
cache_key = f"wp:users:profile:{current_user.id}"

# JTI
redis_key = f"wp:auth:user:{user_id}:access:{access_jti}"
```

### 2.11. Зачем необходимо защищать Redis паролем даже внутри закрытой Docker сети?

**Аргументы:**

1. **Защита от инсайдеров:**
   - В закрытой сети могут быть другие сервисы/разработчики.
   - Пароль предотвращает случайный или злонамеренный доступ.

2. **Защита от lateral movement:**
   - Если злоумышленник проник в один контейнер (например, через уязвимость в приложении), без пароля он сможет прочитать ВЕСЬ кеш Redis.
   - С паролем — барьер усложняет эскалацию привилегий.

3. **Защита от ошибок конфигурации:**
   - Порты могут быть случайно проброшены наружу.
   - Сеть может быть настроена неправильно.
   - Пароль — дополнительный уровень защиты.

4. **Compliance (соответствие стандартам):**
   - Требования безопасности (PCI DSS, GDPR) требуют защиты всех хранилищ данных.
   - Redis по умолчанию не шифрует трафик — пароль минимально защищает от перехвата внутри сети.

5. **Изоляция сред:**
   - В одной Docker-сети могут быть dev, staging, prod сервисы.
   - Пароль разделяет доступ между средами.

**В данной работе:**
```yaml
# docker-compose.yml
command: redis-server --requirepass ${REDIS_PASSWORD} --appendonly yes
```

```python
# app/core/cache.py
Redis(
    host=settings.REDIS_HOST,
    port=settings.REDIS_PORT,
    password=settings.REDIS_PASSWORD or None,
    ...
)
```
